# Релизы и авто-обновления (Sparkle)

Приложение обновляется само через [Sparkle](https://sparkle-project.org). Друзьям
не нужно ничего переустанавливать вручную — при выходе новой версии они увидят
предложение обновиться (или нажмут в меню **«Проверить обновления…»**).

---

## Разовая настройка (один раз)

1. **Создай публичный репозиторий на GitHub**, например `screenshotka`.
2. Открой `sparkle.config` и впиши свой GitHub-логин в `SPARKLE_GITHUB_OWNER`
   (и при желании поменяй `SPARKLE_GITHUB_REPO`). Больше там трогать ничего не нужно.
3. Пересобери приложение: `./build.sh` — теперь в нём зашит правильный URL фида.
4. (Для публикации через одну команду) поставь GitHub CLI и залогинься:
   `brew install gh && gh auth login`.

> 🔑 **Ключи подписи.** Приватный EdDSA-ключ уже создан и лежит в твоём Keychain,
> резервная копия — в `~/Скриншотилка-sparkle-PRIVATE-KEY-backup.txt`.
> **Никому его не давай и не клади в приложение.** Публичный ключ уже вшит в сборку.
> Если потеряешь приватный ключ — friends перестанут получать обновления (придётся
> раздавать новую версию вручную).

> 🛑 **Code signing и macOS-разрешения.** Не пересоздавай сертификат
> `Screenshotka Dev` и не меняй `CODE_SIGN_CERT_SHA1` в `sparkle.config` перед
> обычным обновлением. Разрешения macOS на «Запись экрана», «Камеру» и «Микрофон»
> привязаны к designated requirement приложения, включая certificate leaf.
> Если leaf изменится, пользователи увидят старые галочки в настройках, но новая
> сборка фактически потеряет доступ.
>
> Перед релизом `./build.sh` сам проверяет certificate leaf и падает, если он не
> совпадает с `CODE_SIGN_CERT_SHA1`. Ручная проверка:
>
> ```bash
> codesign -dr - "build/Скриншотилка.app" 2>&1
> ```
>
> В выводе должен быть тот же `certificate leaf`, что указан в `sparkle.config`.

## Первая раздача (initial)

Собери и отдай друзьям первый билд **вручную** (AirDrop / zip):

```bash
./build.sh
# отдать build/Скриншотилка.app (или заархивировать)
```

Поскольку приложение подписано локальным (само-подписанным) сертификатом, при
первом запуске macOS покажет предупреждение Gatekeeper. Друг открывает один раз
через **правый клик → Открыть** (или «Настройки → Конфиденциальность → Открыть
всё равно»). Дальше — как обычное приложение, и все следующие обновления Sparkle
ставит уже без этих плясок.

## Выпуск обновления (каждый раз)

```bash
./release.sh 1.1      # видимая версия 1.1; номер сборки бампится сам
#   или
./release.sh          # та же версия, только +1 к номеру сборки
```

Скрипт соберёт `.app`, упакует в zip, сгенерирует и **подпишет** `appcast.xml`,
и напечатает готовую команду публикации. Останется выполнить (пример):

```bash
gh release create v1.1 \
  dist/Screenshotka-1.1.zip \
  dist/appcast.xml \
  --repo ВАШ_ЛОГИН/screenshotka \
  --title "1.1" --notes "Что нового в 1.1"
```

Всё. В течение суток (или сразу по кнопке «Проверить обновления…») друзья получат
апдейт и поставят его в один клик.

### Как это работает
- `SUFeedURL` в приложении указывает на
  `…/releases/latest/download/appcast.xml` — GitHub всегда отдаёт `appcast.xml`,
  прикреплённый к **последнему** релизу.
- `appcast.xml` ссылается на `Screenshotka-<версия>.zip` из того же релиза и несёт
  EdDSA-подпись архива. Приложение проверяет подпись публичным ключом — поэтому
  подменить обновление нельзя.
- Sparkle сравнивает `CFBundleVersion` (номер сборки). Он **обязан расти** —
  об этом заботится `release.sh`.

## Частые вопросы

**Друг не видит обновление.** Проверь, что в релиз приложены ОБА файла
(`*.zip` и `appcast.xml`), релиз не помечен pre-release, и `SPARKLE_GITHUB_OWNER`
в `sparkle.config` совпадает с реальным репозиторием.

**Хочу нотаризацию (без предупреждения Gatekeeper у друзей).** Нужен платный
Apple Developer ID: подписать `codesign --options runtime` сертификатом Developer ID
и прогнать `xcrun notarytool`. Тогда первый запуск будет без правого клика. Для
круга друзей это необязательно.
